Descripción detallada

La clase CMapParser reune todos los componentes necesarios para el parseo de un mapa.

Es un singleton de inicialización explícita que permite parseo de streams. No obstante incluye métodos que facilitan el parseo de un archivo dado su nombre, o de un string. Al iniciar un nuevo parseo se crea una instancia de CParser y otra de CScanner que se conectan. Luego se alimenta el scanner con la entrada que es un stream y el parser recibe la secuencia de tokens. CMapParser es además transferida al parser para que este pueda añadir las entidades leidas del mapa.

El modo de uso más común será:

Map::CMapParser::Init();
Map::CMapParser *mapParser = Map::CMapParser::getSingletonPtr();
if(mapParser->parseFile("nombre_mapa.extension"))
Map::CMapParser::TEntityList entitylist =
mapParser->getEntityList();
Map::CMapParser::Release();

Hay que tener en cuenta que no se deben hacer cambios en la lista de entidades y es importante llamar a Release al terminar su uso para que se liberen todos los datos creados durante el parseo. Antes de un nuevo parseo la propia clase se encargará de eliminar datos de un parseo previo, por lo que se deben procesar los datos de un mapa ANTES de parsear un nuevo mapa.

Los mapas leidos por este parseador deberán seguir la siguiente estructura:

Map = {
[
EntityName = {
[AttributeName = AttributeValue,]*
},
]*
}

Por ejemplo:

Map = {
gobling1 = {
type = "Gobling",
position = {1.4,0,2.66},
life = 100.0,
...
},
orc1 = {
type = "Orc"
modelname = "orc.mesh",
...
}
...
}

Donde los nombres de los atributos deben empezar por una letra y contener letras, números o '_' y los valores de los atriburos pueden ser cadenas encerradas entre dobles comillas '"', numeros enteros o reales, "true", "false" o posiciones tipo "{x,y,z}". No obstante todos los valores de atributos serán almacenados como strings y será luego, durante el spawn de las entidaes cuando los componentes deban saber que tipo de datos esperan de cada atributo concreto. Los tipos de las entidades deben coincidir con las declaradas en los archivos de blueprints, aunque no se hará ningún tipo de comprobación aquí. Toda entidad declarada deberá como mínimo tener el atributo de nombre "type" y los nombres de las entidades deberán ser diferentes entre sí. Estas comprobaciones solo se realizan en modo Debug, por optimización del código Release. Como curiosidad cabe decir que los saltos de línea, espacios y tabulaciones son opcionales y que puede usarse ';' en vez de ',' como separador si se prefiere.
